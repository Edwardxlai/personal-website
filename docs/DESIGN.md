# 个人网站设计方案

## 0. 项目定义

### 网站目标

这是一个以个人品牌为核心的内容型个人网站。它需要同时服务三类访客：

1. 潜在雇主和合作方，用较短时间判断我的经历、能力和项目质量。
2. 内容受众，找到我在不同平台发布的教程、观点和作品。
3. 同行与朋友，通过网站理解我最近在研究和创造什么。

网站不是简历的网页版本，也不是 React Bits 组件展览。它应该像一张被精心设计、可以向下展开的数字名片。

### 第一版目标

- 首页在 10 秒内说明我是谁、做什么、现在关注什么。
- 作品数量不多时，仍然显得完整、可信、有内容。
- 用一个标志性交互建立记忆点。
- 工作经历、项目、自媒体账号和联系方式形成清晰叙事。
- 支持项目详情和后续教程文章。
- 桌面端有视觉冲击，手机端仍然清晰、快速、可操作。

### 第一版不做

- 不做复杂账号系统。
- 不做评论和社区功能。
- 不做实时同步各平台粉丝数据。
- 不使用多个大型 WebGL 背景。
- 不追求 Kirifuda 式全站 3D 场景。
- 不用空项目卡片填满版面。

## 1. 设计方向

### 视觉命题

**深色编辑式数字名片，使用超大身份排版、冷静的深灰蓝画布、克制的酸性黄绿色，以及一个低强度、持续响应的动态背景。**

整体感觉由三个词约束：

- 精确
- 克制
- 有生命感

它不应该像游戏启动页，也不应该像 SaaS 落地页。访客首先看到的是一个具体的人，其次才是技术和效果。

### 内容叙事

```text
Hero 数字名片
    ↓
Experience 工作经历，先建立可信度
    ↓
Selected Work 精选项目，用少量案例证明能力
    ↓
Content Network 自媒体与内容账号
    ↓
About 个人介绍与当前关注
    ↓
Contact 明确的联系入口
```

作品较少时，工作经历放在项目之前更合适。经历负责建立可信度，项目负责提供证据。等项目达到 4 个以上，并且案例材料足够完整，再考虑将项目提前。

### 交互命题

全站只设置一个一级标志性交互：

> 首屏数字名片会随指针和设备方向产生低幅度空间响应。用户开始滚动后，名片中的身份标识逐渐收束到顶部导航，页面从“认识我”进入“了解我”。

辅助交互只有三类：

- 区块进入时的轻微错位显现。
- 项目图片和文字之间的联动位移。
- 链接、按钮和账号入口的明确反馈。

所有动态效果都必须服从阅读顺序。

## 2. 参考拆解

### Kirifuda

可借鉴：

- 用一个核心视觉隐喻统领全站。
- 首屏不是装饰，而是网站体验本身。
- 黑色画布让视觉对象成为唯一焦点。

不借鉴：

- 全站 WebGL 导航。
- 独立桌面端和手机端实现。
- 大体积纹理、模型和持续渲染。

### 用户参考图

可借鉴：

- 超大姓名快速建立身份。
- 单一高亮色形成记忆点。
- 首页直接暴露工作、项目和联系入口。
- 背景有氛围，但文字仍是主角。

需要调整：

- 不使用大量项目缩略图填满首屏。
- 不使用签名字体装饰，除非提供真实手写签名。
- 不让导航、背景、标题、作品轮播同时争夺注意力。
- 不照搬“左标志、中导航、右按钮”的模板比例。

### 成熟产品方法

- Apple 产品页：每个视口只承担一个主要信息，动效围绕内容推进。
- Linear：深色表面依靠亮度层级而不是厚重阴影建立深度。
- Raycast：高亮色只用于可交互和需要记忆的关键位置。

## 3. 页面架构

### 路由

```text
/                  首页
/work              全部项目
/work/[slug]       项目详情
/notes             文章与教程
/notes/[slug]      文章详情
/about             完整个人介绍，可在第二版上线
/404               定制错误页
```

第一版首页承载完整个人介绍，`/about` 可以暂缓，但项目和文章必须有独立详情页，保证内容可以被分享和搜索。

### 首页区块

#### 01. Hero 数字名片

任务：

- 说明姓名、身份和当前方向。
- 建立视觉记忆点。
- 提供查看经历、作品和联系的明确入口。

建议内容：

```text
[个人标识] 姓名 / English name

职业定位
一句具体的能力描述，不写空泛口号

Based in 城市
Currently 目前状态

View selected work
Contact
```

桌面构图：

- 高度为 `100svh`，最低高度 `720px`。
- 使用 12 列网格。
- 姓名占第 1 至 9 列，位于视口 25% 至 58% 高度区域。
- 职业定位位于姓名下方，最大宽度 `36ch`。
- 当前状态和地点放在右下角，形成信息平衡。
- 导航位于顶部，但不使用完整玻璃卡片包裹。
- React Bits 动态背景铺满画布，内容层与背景层严格分离。

首屏文字草案结构：

```text
ED / 中文姓名

I build useful things,
then explain how they were made.

设计、开发与 AI 工作流
上海 / Available for selected collaborations
```

这里的句子只是结构示例，正式实现前必须替换为真实定位。

动态背景：

- 采用 React Bits 中低频、连续、无明显粒子爆炸的 Shader 类背景。
- 首选方向为 DarkVeil 一类的暗色织物或流体效果。
- 动画速度保持在默认值的 30% 至 45%。
- 背景主色接近画布色，只有局部高光使用品牌色。
- 背景对比度必须保证正文区域始终可读。
- 动画层设置 `pointer-events: none`，交互由独立名片响应层接收。

名片响应：

- 指针移动最大旋转不超过 `2.5deg`。
- 前后景最大位移不超过 `12px`。
- 指针离开后在 `520ms` 内归位。
- 使用 `cubic-bezier(0.16, 1, 0.3, 1)`。
- 手机端不模拟指针悬停，只保留低幅度背景运动。

首屏滚动：

- 滚动前 30% 视口高度时，姓名轻微上移并降低到 88% 透明度。
- 顶部身份标识从 0% 过渡到 100% 透明度。
- 不使用整屏缩放、剧烈模糊或长时间锁定滚动。

#### 02. Experience 工作经历

任务：

- 在项目不多时建立专业可信度。
- 让访客快速理解能力是如何形成的。

布局：

- 不使用普通时间线圆点加竖线。
- 每段经历是一条横向编辑式记录。
- 左侧显示年份和序号，中部显示公司与职位，右侧显示职责和结果。
- 各条记录通过细线和留白分隔，不使用圆角卡片。

桌面结构：

```text
2024-NOW      公司 / 团队
01            职位
              一句话职责
              一项具体结果或代表工作
```

交互：

- 悬停某段经历时，年份和公司名使用品牌色。
- 右侧说明文字向右移动 `6px`。
- 如果经历关联项目，出现 `View related work` 链接。
- 移动端取消悬停动画，内容按年份顺序纵向排列。

内容要求：

- 每段最多 80 至 120 个中文字符。
- 每段必须包含具体职责或结果。
- 不写“负责日常工作”“参与多个项目”一类无法验证的句子。

#### 03. Selected Work 精选项目

任务：

- 用 2 至 3 个真实案例证明能力。
- 让少量作品获得足够展示空间。

项目顺序建议：

1. 最能代表当前方向的项目。
2. 能证明执行能力的真实项目。
3. 这个个人网站本身，作为公开制作案例。

布局：

- 不使用三列等宽卡片网格。
- 每个项目占据一个完整内容段。
- 奇数项目图片偏右，偶数项目图片偏左。
- 项目标题、职责和结果与图片形成交错排版。

每个项目展示：

```text
序号 / 年份
项目名称
一句问题定义
我的职责
技术或方法
一个明确结果
项目详情入口
```

项目图片：

- 桌面比例优先使用 `16:10`。
- 图片圆角 `18px`。
- 图片本身带 `1px` 内描边。
- 不使用统一的悬浮阴影。
- 没有真实封面时使用写明项目名称的设计占位图，不使用随机 AI 图。

交互：

- 图片进入视口时从 `translateY(24px)` 和 `opacity: 0` 进入。
- 项目标题比图片晚 `80ms`。
- 悬停时图片最大放大到 `1.018`。
- 指针位置只影响图片内部高光，不影响整个页面。
- 点击进入独立项目详情页，不使用模态框。

项目详情页结构：

```text
项目标题与摘要
角色、时间、工具
问题背景
设计或开发过程
关键决策
最终结果
复盘
上一篇 / 下一篇
返回项目列表
```

#### 04. Content Network 自媒体矩阵

任务：

- 把内容创作作为作品展示，而不是只放社交图标。
- 解释各平台分别发布什么内容。

标题建议：

- 中文：我在互联网上持续记录什么
- 英文小标题：Content network

布局：

- 使用纵向账号索引，而不是图标网格。
- 每个平台是一整行。
- 左侧为平台名称，中部为内容定位，右侧为代表内容和访问箭头。

示例结构：

```text
Bilibili        网站制作、AI 工作流和完整教程       3 个精选视频  ↗
小红书           设计过程、工具实践和短内容           3 篇精选内容  ↗
公众号           长文教程、复盘和阶段性总结           阅读文章       ↗
GitHub           项目源码与实验                       查看代码       ↗
```

视觉：

- 不直接使用平台品牌色铺满卡片。
- 默认保持全站中性色。
- 悬停时只让平台标识和箭头出现对应品牌色，持续时间 `180ms`。
- 允许每个平台展开显示 1 至 3 个精选内容，但默认不请求平台 API。

数据策略：

- 第一版手动维护精选内容。
- 粉丝数和播放量不是必填内容。
- 如果展示数据，必须标注更新时间。
- 不显示无法长期维护的实时计数器。

#### 05. About 个人介绍

任务：

- 让前面的经历、项目和内容回到一个具体的人。
- 说明当前关注方向与合作偏好。

布局：

- 左侧为真实照片或具有个人意义的静态图像。
- 右侧为两段简短介绍。
- 下方使用关键词行展示关注领域，不使用技能进度条。

内容结构：

```text
我是谁
我如何开始做这些事情

我现在关注什么
我希望参与什么样的项目或合作

Current interests
Astro / Creative coding / AI workflow / Content systems
```

照片要求：

- 优先使用真实人物或工作环境照片。
- 不使用 AI 生成的拟真头像。
- 图片可以是黑白或低饱和度，品牌色仅作为局部覆盖。

#### 06. Contact 联系方式

任务：

- 给浏览过程一个明确结尾。
- 让真正想联系的人不需要继续寻找入口。

布局：

- 使用大字号联系句作为页脚主体。
- 邮箱是主要行动入口。
- 其他平台以文本链接排列。
- 法律信息和版权信息保持低权重。

文案结构：

```text
有合适的项目、内容合作，
或者只是想聊聊？

hello@example.com ↗

GitHub / Bilibili / 小红书 / 微信
```

邮箱交互：

- 悬停时箭头移动 `6px`。
- 点击邮箱正常打开邮件客户端。
- 旁边提供复制按钮，复制成功显示 `Copied`，不使用弹窗。

## 4. 色彩系统

采用深色单主题。第一版不同时开发亮色模式，避免分散设计和测试精力。

所有实现颜色优先使用 OKLCH，并提供十六进制降级值。

| Token | OKLCH | 近似值 | 用途 |
|---|---|---|---|
| `--canvas` | `oklch(0.145 0.012 255)` | `#101318` | 页面主背景 |
| `--surface-1` | `oklch(0.18 0.014 255)` | `#171b21` | 浮层和图片占位 |
| `--surface-2` | `oklch(0.22 0.016 255)` | `#20252c` | 高层级交互表面 |
| `--text-primary` | `oklch(0.94 0.006 110)` | `#eeeee9` | 标题和正文 |
| `--text-secondary` | `oklch(0.72 0.01 255)` | `#a3a9b2` | 说明文字 |
| `--text-tertiary` | `oklch(0.55 0.012 255)` | `#747b85` | 元数据 |
| `--line-subtle` | `oklch(1 0 0 / 0.08)` | `rgba(255,255,255,.08)` | 分隔线 |
| `--accent` | `oklch(0.89 0.215 125)` | `#b7f72a` | 品牌高亮 |
| `--accent-soft` | `oklch(0.89 0.13 125 / 0.18)` | `rgba(183,247,42,.18)` | 背景高光 |
| `--on-accent` | `oklch(0.17 0.025 125)` | `#172006` | 高亮按钮文字 |
| `--danger` | `oklch(0.67 0.19 27)` | `#ef675f` | 表单错误 |

使用规则：

- 画布和深色表面约占 60% 视觉重量。
- 主次文字和图片约占 30%。
- 品牌色不超过 10%，只用于身份、当前状态和关键链接。
- 大段正文不使用品牌色。
- 禁止紫蓝渐变和荧光青色作为第二强调色。
- 品牌色按钮只在 Contact 和关键项目入口使用。

## 5. 字体系统

### 字体选择

品牌关键词是精确、克制、有生命感。

推荐组合：

- 英文展示：`Archivo` Variable，使用较宽字面和 650 至 800 字重。
- 中文与正文：`Source Han Sans SC Variable`。
- 代码：`Maple Mono NF CN`，仅用于代码块和技术标签。

选择原因：

- Archivo 的宽阔结构适合超大姓名排版，不会像常见科技字体那样过于未来化。
- 思源黑体中文质量稳定，字重完整，适合长文和教程。
- Maple Mono 对中英文代码混排友好，但不会进入普通正文。

不使用：

- 不使用 Inter 作为展示字体。
- 不使用装饰性签名字体模拟个人签名。
- 不在同一页面混用两套有强烈个性的展示字体。

### 字号层级

| 层级 | 桌面 | 手机 | 字重 | 行高 | 字距 |
|---|---:|---:|---:|---:|---:|
| Hero name | `clamp(76px, 11vw, 176px)` | `clamp(52px, 17vw, 82px)` | 760 | 0.86 | `-0.045em` |
| Display | `clamp(52px, 7vw, 108px)` | `42px` | 700 | 0.92 | `-0.032em` |
| Section title | `clamp(36px, 4.4vw, 68px)` | `34px` | 650 | 1.0 | `-0.022em` |
| Project title | `clamp(28px, 3vw, 48px)` | `28px` | 620 | 1.05 | `-0.018em` |
| Lead | `22px` | `19px` | 450 | 1.55 | `-0.006em` |
| Body | `17px` | `16px` | 400 | 1.75 | `0` |
| Meta | `13px` | `12px` | 500 | 1.4 | `0.06em` |

规则：

- 标题使用 `text-wrap: balance`。
- 正文使用 `text-wrap: pretty`。
- 正文最大宽度约 `65ch`。
- 英文元数据可以使用大写和正字距，中文元数据不强制增加字距。
- 数字和年份使用 `font-variant-numeric: tabular-nums`。

## 6. 网格与间距

### 页面网格

桌面：

- 最大内容宽度 `1440px`。
- 页面左右安全边距 `clamp(24px, 4vw, 72px)`。
- 12 列网格，列间距 `24px`。

平板：

- 8 列网格。
- 左右边距 `32px`。
- 列间距 `20px`。

手机：

- 4 列网格。
- 左右边距 `20px`。
- 列间距 `12px`。

### 间距尺度

```text
4 / 8 / 12 / 16 / 24 / 32 / 48 / 64 / 96 / 144 / 192
```

区块间距：

- 桌面：`160px` 至 `220px`。
- 平板：`120px` 至 `160px`。
- 手机：`88px` 至 `120px`。

原则：

- 首屏密度高，内容区逐步变得更安静。
- 区块之间依靠留白和标题建立节奏，不通过交替背景色分段。
- 每个区块只承担一个任务。
- 不为了整齐强制所有内容放进卡片。

## 7. 组件规范

### 顶部导航

结构：

```text
[ED · AVAILABLE]               EXPERIENCE  WORK  CONTENT  ABOUT  CONTACT
```

- 高度 `72px`，手机 `60px`。
- 初始状态透明，不添加大面积玻璃背景。
- 滚动超过首屏 70% 后，背景变为 `--canvas` 的 88% 不透明度。
- 底部增加 `1px` 半透明分隔线。
- 当前区块使用品牌色小圆点标记，不使用整块按钮背景。
- 所有目标区域支持锚点滚动。

手机端：

- 左侧保留个人标识。
- 右侧使用 `Menu` 文本按钮，不使用只有三条线的无标签图标。
- 菜单展开为全屏纵向索引。
- 菜单关闭按钮命中区域至少 `44px`。

### 按钮

主按钮：

- 背景 `--accent`。
- 文字 `--on-accent`。
- 高度 `48px`。
- 水平内边距 `22px`。
- 圆角 `999px`。
- 悬停时上移 `-2px`，亮度不增加。
- 按下时 `scale(0.96)`。

次按钮：

- 无填充。
- 使用文字和下划线或箭头。
- 不使用透明描边胶囊作为默认次按钮。

### 圆角系统

```text
--radius-sm: 6px
--radius-md: 12px
--radius-lg: 18px
--radius-xl: 28px
--radius-pill: 999px
```

- 图片和大型媒体使用 `18px`。
- 小型状态标签使用胶囊。
- 正文区块默认无圆角容器。

### 图标

- 统一使用 Lucide。
- 普通尺寸 `18px`。
- 线宽 `1.75`。
- 外链统一使用 `ArrowUpRight`。
- 图标按钮必须提供 `aria-label`。
- 不混用 Heroicons、Font Awesome 和平台风格不统一的线性图标。

## 8. 动效规范

### 动效层级

#### 一级：首屏标志性交互

- React Bits Shader 背景。
- 数字名片低幅度倾斜。
- 身份标识随滚动收束至导航。

#### 二级：内容结构动画

- 区块标题进入。
- 项目图片与文字错时出现。
- Experience 行按顺序出现。

#### 三级：反馈动画

- 链接箭头位移。
- 按钮按压缩放。
- 社交平台颜色显现。
- 图片轻微缩放。

### 时间和缓动

```text
快速反馈：120ms
普通悬停：180ms
内容进入：520ms
首屏归位：520ms
大型转场：720ms
主缓动：cubic-bezier(0.16, 1, 0.3, 1)
退出缓动：cubic-bezier(0.4, 0, 1, 1)
```

禁止：

- 不使用弹簧回弹和橡皮筋效果。
- 不使用 `transition: all`。
- 不动画 `width`、`height`、`margin` 和 `padding`。
- 不让每个标题逐字播放超过 1 秒。
- 不劫持浏览器滚动。

### React Bits 使用边界

第一版最多使用 3 个 React Bits 组件：

1. Hero 背景组件。
2. Hero 主标题或身份标识的轻量文字显现。
3. 一个项目媒体区域的指针响应效果。

每个组件接入前必须完成：

- 替换为本项目颜色。
- 降低默认运动强度。
- 检查包体积和依赖。
- 添加移动端简化模式。
- 添加 `prefers-reduced-motion` 降级。
- 确认关闭组件后内容仍然完整。

Astro 加载策略：

- 首屏背景：`client:load`。
- 首屏外的项目交互：`client:visible`。
- 非关键增强：`client:idle`。
- 完全依赖浏览器 API 的组件：`client:only="react"`。

## 9. 深度与表面

深色主题主要依靠亮度层级建立深度：

```text
Canvas      --canvas
Surface 1   rgba(255,255,255,0.02)
Surface 2   rgba(255,255,255,0.04)
Prominent   rgba(255,255,255,0.055)
```

规则：

- 不在深色背景使用传统黑色大阴影。
- 分隔优先使用留白，其次使用 `1px` 半透明线。
- 图片使用内描边，避免边缘融入背景。
- 只有导航和少量浮动控件允许使用半透明表面。
- 不使用全站玻璃拟态。

## 10. 响应式行为

### 断点

```text
Mobile:   0-639px
Tablet:   640-1023px
Desktop:  1024-1439px
Wide:     1440px+
```

### 手机端调整

- Hero 高度使用 `100svh`，处理浏览器地址栏变化。
- 姓名最多分成两行，不让单词被随机截断。
- 动态背景速度降低 40%。
- 禁用指针倾斜和磁吸效果。
- Experience 从三列行变成单列信息块。
- 项目统一为图片在上、文字在下。
- 社交账号每行最少高度 `72px`。
- Contact 邮箱允许自动换行。
- 所有交互目标至少 `44 × 44px`。
- 使用 `env(safe-area-inset-*)` 处理刘海和底部手势区域。

### 触摸设备

- 所有 hover 效果放进 `@media (hover: hover)`。
- 不通过 hover 才能看到必要信息。
- 使用 `touch-action: manipulation`。
- 不让项目图片跟随手指持续旋转。

## 11. 性能与无障碍

### 性能预算

第一版目标：

- 首屏 JavaScript 压缩后不超过 `180KB`，大型 Shader 依赖单独分包。
- 首屏背景资源不超过 `1.5MB`。
- 首屏不自动加载项目详情图片。
- 项目图片使用 AVIF 或 WebP。
- 所有图片提供明确宽高。
- 字体首屏最多预加载 2 个文件。

如果 React Bits 某个组件引入大型 Three.js 或 OGL 依赖，却只产生轻微装饰效果，应当换成 CSS 或更轻的 Canvas 实现。

### 动态效果降级

`prefers-reduced-motion: reduce` 时：

- 关闭首屏指针倾斜。
- 背景停止或降为极慢运动。
- 内容直接显示，不执行错时进入。
- 保留颜色、对比和静态层次。

WebGL 初始化失败时：

- 显示静态深色背景和品牌色柔光图。
- 所有文字和导航保持正常。
- 不显示阻塞式错误。

### 无障碍

- 页面第一个可聚焦元素是 `Skip to content`。
- 使用语义化的 `header`、`nav`、`main`、`section`、`article`、`footer`。
- 每个 section 有可访问标题。
- 焦点状态使用品牌色 `2px` 外轮廓。
- 正文与背景对比度达到 WCAG AA。
- 动态背景标记为装饰，不进入辅助技术阅读顺序。
- 导航、项目和联系方式都能通过键盘完成。

## 15. 设计守则

### 应该做

- 让真实内容决定版面大小。
- 用一个动态背景建立记忆点。
- 给少量项目更大的叙事空间。
- 把自媒体内容当作正式作品。
- 在手机端主动简化动态效果。
- 每次加入特效前先确认它服务哪个信息。

### 不应该做

- 不把首页做成 React Bits 组件合集。
- 不用九宫格填充不存在的作品。
- 不使用紫蓝渐变和泛滥的玻璃拟态。
- 不使用技能百分比进度条。
- 不在每个区块放两个并排 CTA。
- 不自动播放有声音的视频。
- 不为了动画锁定滚动或延迟页面可用时间。
- 不把 About 和 Contact 混成一个含糊区块。

## 16. 验收标准

### 首屏

- 5 秒内能读出姓名、职业定位和当前方向。
- 动态背景不会降低文字可读性。
- 首屏只有一个主要视觉焦点。
- CTA 不超过两个。

### 内容

- 工作经历每段有具体职责或结果。
- 项目数量少，但每个项目信息完整。
- 自媒体区说明“发布什么”，不只展示平台 Logo。
- About 能解释个人动机和当前关注。

### 视觉

- 全站只使用一个强调色。
- 没有三个同样的圆角阴影卡片连续出现。
- 标题、正文和元数据层级清楚。
- 动效关闭后，页面仍然成立。

### 技术

- React 只存在于确实需要交互的区域。
- 首屏外互动按需加载。
- 支持 Reduced Motion。
- 图片尺寸固定，无明显布局偏移。
- 自定义 404、返回路径和键盘焦点完整。

### Content network

### Contact

> 创建大字号联系页脚，背景保持 `--canvas`，顶部留白 180px。联系句使用 `clamp(48px, 7vw, 104px)`，行高 0.96，字距 `-0.032em`。邮箱作为主要入口，使用 `--accent`，箭头悬停移动 6px。提供复制邮箱按钮，成功状态使用行内文字 `Copied`，不使用 toast 或模态框。
